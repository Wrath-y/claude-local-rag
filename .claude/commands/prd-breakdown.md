# PRD Breakdown

你是需求分析专家和系统架构师。任务是读取 PRD 文档（支持网页 URL、本地文件、直接粘贴文本），首先将其拆分为独立的可实施需求，**其次输出技术方案文档**，最后再逐个生成 openspec 开发计划，最终由用户决定如何实施。

## 入参解析

从 `{{args}}` 中提取文档来源，按以下规则判断类型：

| 输入形式 | 判断依据 | 处理方式 |
|---------|---------|---------|
| 网页 URL | 以 `http://` 或 `https://` 开头 | 使用 WebFetch 工具抓取 |
| 本地文件路径 | 以 `/` 或 `./` 开头，或包含文件扩展名 | 使用 Read 工具读取 |
| 直接粘贴文本 | 非 URL、非路径的多行文本 | 直接作为 PRD 内容使用 |

**参数为空时**，停止并提示：

```
用法：/prd-breakdown <来源>

支持的来源类型：
  网页 URL：https://example.com/docs/prd 或 https://xxx.feishu.cn/docx/AbCdEf123456
  本地文件：/path/to/prd.md 或 ./docs/prd.txt
  直接内容：将 PRD 文本直接粘贴作为参数
```

---

## 执行流程

### 第一阶段：获取 PRD 内容

根据入参类型，选择对应方式获取内容：

**网页 URL**：使用 WebFetch 工具抓取页面内容（包含飞书、Notion 等文档链接，均按普通网页处理）。若抓取失败，停止并提示用户检查 URL 是否可公开访问。

**本地文件路径**：使用 Read 工具读取文件内容。支持 `.md`、`.txt`、`.pdf` 等格式。若文件不存在，停止并提示用户确认路径。

**直接粘贴文本**：将 `{{args}}` 内容直接作为 PRD 使用，无需额外获取步骤。

获取内容后，若内容为空或明显不是 PRD（少于 100 字），停止并提示用户重新提供。

---

### 第二阶段：分析并拆分需求

仔细阅读 PRD 全文，按以下原则拆分需求：

**拆分原则：**

1. **功能独立**：每个需求是一个可独立开发、测试和上线的功能单元
2. **粒度适中**：不宜过细（避免一个功能拆成十个小点），也不宜过大（避免多个不相关功能合并）
3. **前后端分离**：若 PRD 明确区分前后端，保持对应关系，后端需求拆分给本项目
4. **按优先级/模块分组**：尊重 PRD 原有的章节结构和优先级

**每个需求提取：**

- `name`：kebab-case 英文标识（如 `add-payment-popup`）
- `title`：简短中文标题（15 字以内）
- `summary`：关键信息，包含：目标用户/场景、核心功能点、接口或数据变更（2-4 行）
- `module`：所属模块（如：订单、支付、用户、后台管理等）
- `priority`：HIGH / MEDIUM / LOW（根据 PRD 描述判断）

拆分完成后，**向用户展示所有候选需求并询问是否调整**：

```text
📋 初步拆分完成（共 N 个需求）

┌─────────────────────────────────────────────────────
│ #1 [HIGH] add-payment-popup - 新增支付弹窗
│    模块：支付
│    摘要：为用户充值流程增加确认弹窗，支持7日免打扰。
│         需新增接口：POST /api/v1/popup/dismiss
│         涉及表：user_popup_settings（新增）
├─────────────────────────────────────────────────────
│ #2 [MEDIUM] order-status-filter - 订单列表状态筛选
│    模块：订单
│    摘要：后台订单列表支持按支付状态多选筛选。
│         修改接口：POST /admin/v1/orders（新增 status[] 参数）
├─────────────────────────────────────────────────────
...

以上需求是否需要调整（合并/拆分/剔除）？
直接回复"确认"继续，或告知调整意见：
```

**输出后停止，等待用户回复。**

根据用户反馈调整需求列表后（或用户直接确认），进入第三阶段。

---

### 第三阶段：写入需求文件

**3.1 生成文件名**

- 标题 `RAG系统 1.0.0` → `rag-system-v1.0.0.md`
- 无法识别时使用 `prd-<today>.md`


### 第五阶段：逐个生成 openspec 计划

根据用户选择，**依次**为每个确认的需求生成 openspec 计划。

**5.1 告知当前进度并请求确认**

展示当前需求的详细信息，并等待用户确认后再生成：

```text
⚙️  准备生成需求 #N 的 openspec 计划（N/M）

Change 名称：<name>
标题：<title>
摘要：<summary>

确认生成？（直接回复"确认"或"y"继续，回复"跳过"跳过此需求）
```

**输出后停止，等待用户回复。**

用户确认后继续；用户回复跳过则标记为 ⏭️ 已跳过，进入下一个需求。

**5.2 调用 openspec-propose**

使用 **Skill 工具** 调用 `openspec-propose`，传入需求描述：

- skill: `openspec-propose`
- 描述内容：基于需求的 `name`、`title`、`summary`，组织为清晰的一段话

示例格式：

```text
Change name: <name>

<title>

<summary 展开为完整描述，包含背景、目标、接口/数据变更要点>
```

**5.3 等待 openspec-propose 完成**

openspec-propose 会自动创建 proposal.md、design.md、tasks.md 三个文件。完成后：

- 将 `docs/prd/<filename>.md` 中对应需求的 openspec 状态从 `⏳ 待生成` 更新为 `✅ 已生成`
- 继续处理下一个需求

**5.4 如遇到 openspec-propose 需要交互询问**

如果 openspec-propose 在生成过程中向用户提问，正常响应用户的回答后继续流程。

## 注意事项

- **只处理后端需求**：若 PRD 涉及前端/客户端部分，在摘要中注明，但标记为本项目不实施
- **不自动执行实施**：本命令只生成计划，不修改任何业务代码
- **需求编号从 1 开始**，便于用户引用
- **遇到模糊需求**：在 summary 中标注"⚠️ 需澄清：<具体问题>"，不阻塞流程
- **PRD 中的 UI 细节**（如颜色、动画）不需要纳入后端需求，可忽略
